### 2.2 Ключевые связи

- **app.js → API**: прямой `fetch()` к `api_save.php`, `api_gantt.php`, `api_projects.php`, `api_update_task.php`, `api_update_input.php`, `api_upload_csv.php`. Без прослойки, без стандартного формата ответа.
- **API → БД**: все эндпоинты напрямую вызывают `DB::getConnection()` (синглтон PDO). Никакого репозиторного слоя или ORM.
- **PHP-страницы → header/footer**: `index.php`, `calc.php`, `toc.php`, `report.php` подключают `header.php`/`footer.php` (общий шаблон).
- **Два параллельных UI-kit**: `frontend/ui-kit/` и `prikidka_pro/frontend/ui-kit/` — **идентичные копии**, 100% дублирование (10 файлов × 2).
- **prikidka vs prikidka_pro**: `prikidka_pro` — рефакторинг `prikidka`, но оба существуют в проекте одновременно. `prikidka` — монолит на 800 строк в `index.php`, `prikidka_pro` — модульная архитектура с разделением JS.
- **src/** (TocCalculator, TaskValidator, ResourceCsvParser): написаны качественно, с классами, но **не используются** эндпоинтами — API дублирует логику прямо в своих файлах.

```php

$port = '3308';
$db   = 'olegcherne_toc';
$user = 'olegcherne_toc';
$pass = 'eR#RCBR5E3ASKWSD';
```

**Рекомендация:** Немедленно сменить пароль на production-сервере. Удалить `db.php` и `config.php` из Git-истории (`git filter-branch` / `BFG Repo-Cleaner`). Создать `.gitignore`.

### 3.2 XSS-уязвимости (Cross-Site Scripting)

**Файл:** `public_html/app.js`, строки ~250, 256, 761, 791

Пользовательский ввод (названия задач, имена ресурсов) вставляется напрямую через `innerHTML`:

```js
// Строка 250 — пользовательское имя задачи напрямую в innerHTML
row.innerHTML = `<td>${task.name}</td>...`;

// Строка 761 — имя ресурса из пользовательского ввода
el.innerHTML = resource.name;
```

**Вектор атаки:** если злоумышленник сохранит задачу с именем `<img src=x onerror=alert(document.cookie)>`, этот скрипт выполнится у всех пользователей, открывших проект.

### 3.4 Отсутствие валидации входных данных в API

**Файл:** `api_save.php`, строки 33–34

```php
$stmt->execute([
    $task['id'], $task['name'], $task['start'], $task['end'],
    $task['resource'] ?? '', $task['utilization'] ?? '0',
    // ...
]);
```

- Поле `$task['id']` не валидируется — может содержать SQL-инъекцию при определённых конфигурациях PDO (хотя prepared statements снижают риск).
- `$task['start']` / `$task['end']` не проверяются на формат даты — в БД записывается любая строка.
- `utilization` не проверяется на числовой диапазон.

### 3.7 Раскрытие деталей БД при ошибке подключения

**Файл:** `public_html/db.php`, строка 20

`PDOException::getMessage()` раскрывает: DSN, хост, порт, имя БД, пользователя. В production это даёт злоумышленнику полную карту инфраструктуры при любом сбое MySQL.

### 3.8 Null-guards: ~20+ небезопасных обращений к DOM

**Файл:** `public_html/app.js`

Повсеместное обращение к `document.getElementById()` и `querySelector()` без проверки на `null`:

```js
// Типичный паттерн в app.js:

// Если элемент не найден — Uncaught TypeError: Cannot set properties of null
```

Примеры строк: ~100, 150, 280, 350, 420, 574, 660, 896, 914 и многие другие. При динамическом DOM (исчезновение элементов при переключении режимов) это приводит к падению всего JS.

### 3.9 Неперехваченные ошибки fetch

**Файл:** `public_html/app.js`

Все `fetch()`-вызовы (к `api_save.php`, `api_gantt.php` и др.) выполняются без `.catch()`:

```js
fetch('api_save.php', { method: 'POST', body: JSON.stringify(data) })
    .then(r => r.json())
    .then(d => { /* ... */ });
// НЕТ .catch() — сетевая ошибка проглатывается молча
```

## 6. Утечки памяти и ресурсов

### 6.1 app.js — незакрытые Event Listeners

**Файл:** `public_html/app.js`, строки 686–704, 1044–1084

Каждый вызов `renderTaskTable()` создаёт новые `addEventListener` через `querySelectorAll`, при этом предыдущие DOM-элементы удаляются через `innerHTML = html`. Старые слушатели остаются в памяти:

```js
// Строка 1042 — замена innerHTML уничтожает DOM, но не снимает слушатели:
tbody.innerHTML = html;
// Строки 1044-1084 — тут же навешиваются НОВЫЕ слушатели:
tbody.querySelectorAll('.inline-name-input').forEach(el =>
    el.addEventListener('change', e => { /* ... */ })
);
```

При активной работе с проектом (перетаскивание задач, переключение режимов) утечка растёт линейно: ~10 слушателей × N задач × количество ререндеров.

---

### 6.2 Gantt — повторное создание без destroy()

**Файл:** `public_html/app.js`

При вызове обновления диаграммы Gantt создаётся новый экземпляр без вызова `destroy()` на старом. Старый canvas остаётся в памяти.

---

### 6.3 Scroll listener без снятия

**Файл:** `public_html/app.js`

Глобальный `scroll` listener, добавленный через `window.addEventListener('scroll', ...)`, не снимается при уходе со страницы (SPA-навигация).

## 8. Проблемы с БД

### 8.1 Runtime-миграция в API

**Файл:** `public_html/api_save.php`, строка 16

```php
$pdo->exec("ALTER TABLE tasks ADD COLUMN is_buffer TINYINT(1) DEFAULT 0");
```

Миграция схемы БД выполняется при каждом сохранении проекта. Это антипаттерн: DDL-операции не должны выполняться в рантайме API-запроса.

### 8.2 DELETE ALL + INSERT ALL — отсутствие транзакций

**Файл:** `public_html/api_save.php`, строки 22–28

```php
$pdo->exec("DELETE FROM tasks");
foreach ($tasks as $task) {
    $stmt->execute([...]);
}
```

Если вставка упадёт на середине, БД останется в несогласованном состоянии (часть задач удалена, часть — нет). Не обёрнуто в `BEGIN/COMMIT`.

### 8.3 Две конфигурации БД

- `db.php` (основная) — жёстко закодированные учётные данные production.
- `db_config.php` — локальная конфигурация (root, пустой пароль).
